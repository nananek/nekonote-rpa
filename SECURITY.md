# Security Policy

## サポート対象バージョン

セキュリティアップデートは **最新リリース** のみを対象とします。
旧バージョンをお使いの場合は、最新版へのアップデートを推奨します。

## 脆弱性の報告

セキュリティ上の問題を発見した場合は、**GitHub Private Vulnerability Reporting** を通じてご報告ください。

1. リポジトリの **Security** タブ → **Report a vulnerability** を選択
2. 脆弱性の詳細、再現手順、影響範囲を記載

公開 Issue での脆弱性報告は **お控えください**。報告を受領後、可能な限り迅速に確認・対応いたします。

## サプライチェーンセキュリティ方針

Nekonote はエンドユーザーの Windows マシン上で動作する RPA ツールです。
依存関係の侵害がユーザー環境に直接影響するため、以下の対策を実施しています。

### 依存関係の管理

| 領域 | 対策 |
|------|------|
| **npm** | `package-lock.json` による完全なロック。CI では `npm ci` で再現可能なインストールを保証 |
| **Python (pip)** | `pyproject.toml` で最小バージョンを指定。Dependabot が週次で更新を監視 |
| **git 依存** | コミット SHA に固定し、上流の変更が意図せず混入することを防止 |
| **Dependabot** | npm / pip / GitHub Actions の 3 エコシステムを週次で監視 |

### CI/CD パイプラインの保護

- **GitHub Actions のコミット SHA ピン留め**: 全アクションをタグではなくコミット SHA で指定し、タグの差し替えによるサプライチェーン攻撃を防止
- **最小権限の原則**: ワークフローの `permissions` を必要最小限に制限
- **依存関係の脆弱性スキャン**: CI で `npm audit` および `pip-audit` を実行し、既知の脆弱性を含む依存を検出
- **静的解析**: CodeQL による JavaScript/TypeScript・Python の週次セキュリティスキャン

### リリースプロセス

- リリースビルドは GitHub Actions の管理するランナー上で実行（ローカルビルドは配布しない）
- 成果物は GitHub Releases に直接アップロードされ、改ざんリスクを低減
- 自動更新 (electron-updater) は GitHub Releases から取得し、electron-updater の検証機構で整合性を確認

### 既知の制限

- **コード署名**: 現時点ではインストーラー・実行ファイルにコード署名を適用していません。将来的な対応を検討中です
- **Python ロックファイル**: pip は完全なロックファイル機構を標準では持たないため、Dependabot による監視と最小バージョン指定で補完しています
